# TempHumidityPlotter Skills

## 目标

在这个项目里，优先延续现有的轻量配置驱动开发方式：

- 用 `common.env` 管理本机相关的运行参数。
- 用 `config.yaml` 作为应用级配置入口，并通过环境变量注入覆盖默认值。
- 用 `logging_config.py` 统一初始化日志，不在各个脚本里重复造日志配置。
- 保持 Python 项目的常规忽略规则和当前的 `flake8` 风格约束。

## 当前基础配置

### `common.env`

本地运行参数放在这里，当前已包含：

- `LOG_LEVEL`
- `INPUT_FILE`
- `OUTPUT_PATH`

约定：

- 这里可以放机器相关、路径相关、调试相关配置。
- 不要把这些值硬编码进业务脚本。
- 绝对路径只应出现在本地环境文件里，不应扩散到源码里。

### `config.yaml`

应用配置统一从这里读取，当前 `app` 节点包含：

- `log_level`
- `input_path`
- `output_dir`
- `input_file_type`

约定：

- 默认值写在 YAML 里。
- 本地差异通过 `${ENV_VAR:-default}` 形式覆盖。
- 新增配置项时，优先继续放在 `app` 节点下，保持入口集中。
- 如果新增的是本地敏感配置，先在 YAML 里定义环境变量占位，再放到 `.env` 类文件中赋值。

### `logging_config.py`

项目已经有统一日志助手，后续脚本应直接复用：

- `setup_logger(...)` 用于初始化根日志。
- `get_logger(name)` 用于按模块获取 logger。
- 默认日志文件放在 `log/` 目录下，并基于启动脚本名生成文件名。
- 日志会同时输出到控制台和滚动文件，单文件上限 10 MB，保留 5 份备份。

约定：

- 新增入口脚本时，在启动早期调用 `setup_logger(...)`。
- 模块内统一用 `get_logger(__name__)` 获取日志对象。
- 除非有明确理由，不要在其他文件里重新配置 handler、formatter 或 root logger。

推荐模式：

```python
import logging

from logging_config import get_logger, setup_logger

setup_logger(log_level=logging.INFO)
logger = get_logger(__name__)
```

### `.flake8`

当前风格基线：

- 最大行宽 `120`
- 忽略 `BLE001`
- 忽略 `W503`
- 忽略 `W504`

约定：

- 新代码默认遵守 `flake8`。
- 允许保持当前项目偏宽松的异常处理风格，但不要滥用；能收窄异常类型时仍然优先收窄。
- 不要为了压缩行宽牺牲可读性；120 列是当前项目明确允许的上限。

### `.gitignore`

当前忽略策略已经覆盖：

- `output/`、`input/`、`log/` 等运行产物目录
- `.conda/`、`.venv/`、`env/` 等本地环境目录
- `common.env` 等本地环境文件
- Python 缓存、测试缓存、打包产物和常见工具缓存

约定：

- 生成型文件优先写入已忽略目录，比如 `output/`、`log/`。
- 新增本地专用配置文件时，同步补进 `.gitignore`。

## 开发技能

### 新增脚本时

遵循下面的接线顺序：

1. 从环境变量或 `config.yaml` 读取输入输出路径，不直接写死。
2. 启动时初始化日志。
3. 把产出写到 `output/`，把日志写到 `log/`。
4. 把可变配置做成环境变量覆盖项，而不是散落在代码常量里。

### 新增配置时

优先按这个模式扩展：

1. 在 `config.yaml` 增加默认配置。
2. 需要本地覆盖时，给它绑定环境变量占位。
3. 在 `common.env` 中补充本机值。
4. 在代码里统一从配置层读取，不绕过配置文件直接读硬编码常量。

示例：

```yaml
app:
  sample_rate: ${SAMPLE_RATE:-60}
```

### 新增日志时

保持日志信息对排查问题有帮助：

- 记录输入路径、输出目录、关键参数和阶段性进度。
- 出错时记录上下文，不只打印一句失败。
- 不在循环里无节制打印高频日志，避免刷屏和撑爆日志文件。

### 处理本地路径时

这个项目已经出现了 Windows 绝对路径输入，因此后续代码应注意：

- 内部处理尽量使用 `pathlib.Path`。
- 允许读取绝对路径，也要兼容相对路径。
- 目录不存在时优先自动创建输出目录。

## 默认工作方式

如果后续要继续在这个项目上写功能，默认按下面的思路推进：

- 先补齐或复用 `common.env` 和 `config.yaml`，再写业务代码。
- 先复用 `logging_config.py`，不要从零搭日志系统。
- 新文件尽量保持单一职责：配置、日志、业务逻辑分开。
- 运行产物不入库，本地差异不入库。

## 对话与执行约束

从本条规则生效后，后续每次对话默认遵循以下流程：

1. 用户提出需求或想法后，先复述当前理解。
2. 主动询问为避免误解所需的关键细节（输入、输出、边界、验收标准等）。
3. 在用户明确同意开始实现前，只进行需求澄清，不编写代码、不改文件、不执行实现型修改。
4. 仅在收到用户许可后进入编码与修改阶段。

执行口令：

- 默认状态为“先澄清，后实现”。
- 若用户在同一轮消息中明确授权“可以直接写代码”，则视为已许可执行。

## 适用场景

这份技能文档适用于：

- 新建温湿度数据处理脚本
- 扩展输入文件类型
- 增加导出目录、日志级别、运行参数等配置项
- 规范化项目结构，避免把本地实验代码直接写成不可复用脚本

## 不建议的做法

- 在源码里写死本机绝对路径
- 每个脚本各自维护一套日志配置
- 把运行输出、日志文件、虚拟环境提交到版本库
- 配置项同时散落在 `.env`、YAML 和 Python 常量里，导致优先级不清晰
